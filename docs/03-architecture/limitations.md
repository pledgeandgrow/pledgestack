# PledgeStack — Current Limitations

## Build & Bundling

- **PledgePack binary required** — The Rust binary (`pledgepack@^0.1.8`) is installed from npm via postinstall. If the binary download fails (network issues, unsupported platform), builds will not work. The framework falls back to Node.js-based builds in this case.

## Rendering

- **SSR runs in Node.js** — `renderToPipeableStream` is piped directly for streaming SSR, but the React rendering pipeline executes in Node.js (V8), not in the Rust binary. Rust SSR handles static HTML segments extracted at build time — dynamic React rendering still runs in V8. The Rust native addons (`rust-ssr`, `rust-html`, `rust-dom-renderer`) provide native rendering for static parts with automatic JS fallback.
- **RSC payload generated by Node.js** — RSC streaming and client manifests work via `renderRSCToHTML`, but the RSC payload is generated by Node.js. The Rust dev server handles static assets and HMR; RSC requests are proxied to a Node.js process. The `rust-rsc` native addon provides flight serialization with swc-based module analysis, with React Server DOM fallback.
- **Native addon availability** — All 8 Rust native addon crates (`rust-html`, `rust-ssr`, `rust-rsc`, `rust-html-transformer`, `rust-dom-renderer`, `rust-rsc-deserializer`, `rust-ssr-profiler`, `rust-hydration`) have JS fallbacks. When `.node` files are not compiled, the framework automatically falls back to JavaScript implementations. No errors are thrown — functionality is preserved with JS performance.
- **No `use client` / `use server` directive parsing** — PledgeStack uses the Pledge System (`pledge()` HOC) instead of file-level directives. This is by design, not a limitation.

## Data & State

- **Client-side data hooks implemented** — `useFetch`, `useSWR`, and `useMutation` are available from `pledgestack/client` with caching, revalidation, and optimistic updates. SWR/React Query integration is no longer needed for basic use cases.

## Server Lifecycle

- **Instrumentation wired** — `loadInstrumentation()` is called during `startNodeServer()` and `createEdgeHandler()` startup. It loads `instrumentation.ts` from the app root and calls `register()` before any requests are handled.
- **Static export wired** — `generateStaticExport` is called during `pledge build` when `config.output === 'export'`. It pre-renders all routes (including dynamic routes via `generateStaticParams`) to static HTML files.

## Developer Experience

- **Storybook integration** — `pledge storybook` command with zero-config setup using PledgePack as builder, auto-discover `*.stories.tsx` files.
- **Streaming metadata** — Inject `<title>` and `<meta>` tags via streaming HTML transformer that patches `<head>` after initial flush, async `generateMetadata()` without blocking first byte.
- **Generated route types** — Auto-generates `__pledge_route_types.d.ts` from file-based router at build time: typed `params`, `searchParams`, layout chain types, route metadata.

## Deployment

- **Rust production server not yet available on all platforms** — `pledgepack serve` (Rust HTTP server) requires the native binary. If unavailable, production server falls back to Node.js.

## Framework Self-Build

- **esbuild for framework bundling** — The framework itself uses esbuild (not PledgePack) to bundle the CLI package for npm publish. PledgePack is used to bundle user apps, not the framework itself. This is by design — same pattern as Next.js using webpack/turbopack for user apps while building Next.js itself with its own tooling.

## PSX / PS Format (Rust Integration)

- **Rust toolchain required for .psx/.ps** — The `cargo` command and Rust toolchain must be installed to compile `.psx` and `.ps` files. If absent, `.psx` files still work as pure TSX (Rust calls throw a helpful error), and `.ps` files export a stub that throws on call.
- **Slow Rust compile times** — First build of a `.psx`/`.ps` file takes 30-60 seconds (downloads + compiles all crates). Incremental builds take 2-10 seconds. Compare: esbuild transforms `.tsx` in 50ms. Dev iteration is slower for Rust edits.
- **HMR for Rust code** — PSX HMR is implemented with incremental `cargo build`, content-hash change detection, and module-level invalidation. Rust code changes trigger recompilation but preserve TSX state across recompiles.
- **Rust knowledge required** — Writing `.psx`/`.ps` files requires Rust knowledge (ownership, lifetimes, borrowing). Most web developers know TypeScript, not Rust. This is an adoption barrier for teams.
- **Smaller Rust web ecosystem** — No Prisma GUI, no Stripe SDK, no Auth0/NextAuth equivalent, no SendGrid/Twilio SDKs in Rust. Common SaaS integrations require more glue code than Node.js. PSX integrations bridge this gap with JS fallbacks using npm packages.
- **Debugging across JS→Rust boundary** — Stack traces don't cross the NAPI boundary cleanly. Source maps map Rust code positions back to `.psx`/`.ps` source lines. `println!` output is captured and redirected to `console.log` with source attribution.
- **IDE support for .psx** — A VS Code extension exists with language configuration and syntax highlighting. Full IntelliSense for Rust inside `<rust>` blocks is planned.
- **Two toolchains** — `package.json` + `npm` for JS deps, `Cargo.toml` + `cargo` for Rust deps. Two lockfiles, two update processes, two security audit processes.
- **Windows development rougher** — Rust on Windows requires MSVC build tools. Some crates don't compile cleanly on Windows. `cargo` is slower on Windows than macOS/Linux.
- **Regex-based parser** — The `.psx` parser uses regex, not a real Rust AST parser. Edge cases in complex Rust syntax may not parse correctly. This will be replaced with a syn-based parser.
- **PSX integrations have JS fallbacks** — All 15 PSX integration classes (SQLx, Redis, Auth, Image, PDF, Jobs, Cron, Email, HTTP, WebSocket, File, Tracing, Crypto, ML) gracefully fall back to Node.js packages when native Rust addons are unavailable. This ensures integrations are usable without Rust installed.
- **Native addon tests** — 80+ tests cover all render modules and PSX integrations using Vitest, verifying fallback behavior and core functionality.
