# PledgeStack ‚Äî Not Needed / Out of Scope

Things we are explicitly **not** building. These are either solved by the ecosystem, unnecessary for our architecture, or actively harmful to our goals.

## PledgePack Owns These (Not PledgeStack)

These are build/bundler/runtime concerns handled by [PledgePack](https://www.npmjs.com/package/pledgepack). PledgeStack wraps them but does not implement them:

- **Bundling** ‚Äî Tree-shaking, code splitting, minification, source maps are PledgePack's job
- **CSS processing** ‚Äî Tailwind compilation, CSS code splitting, critical CSS inlining are PledgePack's job
- **Asset pipeline** ‚Äî Image/font optimization, content hashing, CDN-ready URLs are PledgePack's job
- **Dev server** ‚Äî File watching, HMR, on-the-fly compilation are PledgePack's job
- **Production server** ‚Äî Rust HTTP server (Axum/Hyper), static file serving, gzip/brotli compression are PledgePack's job
- **Build cache** ‚Äî Content-hash cache, remote cache, incremental builds are PledgePack's job
- **Bundle analysis** ‚Äî `pledge analyze` treemap is PledgePack's job
- **Service worker** ‚Äî Runtime caching strategies are a PledgePack plugin concern
- **HTTP/2 push** ‚Äî Server-level feature handled by PledgePack's production server
- **SRI generation** ‚Äî `integrity` attributes for scripts are generated by PledgePack at build time

## We Don't Need

- **Custom package manager** ‚Äî pnpm is excellent. We use it, we don't replace it.
- **Custom TypeScript compiler** ‚Äî `tsc` or `swc` handles this. PledgePack may use `oxc` for parsing but not full TS type checking.
- **Custom CSS engine** ‚Äî Tailwind v4 + native CSS is enough. No custom CSS-in-JS runtime.
- **Custom test runner** ‚Äî Use `vitest` or `node:test`. Don't reinvent.
- **Custom linter** ‚Äî Use ESLint. We'll write a plugin, not a replacement.
- **Custom formatter** ‚Äî Use Prettier. Don't reinvent.
- **Custom VS Code language server** ‚Äî TypeScript language server already works. We add an extension for file icons and config IntelliSense, not a language server.
- **GraphQL layer** ‚Äî Not opinionated about data fetching. Use `fetch()`, tRPC, or whatever. We provide `cachedFetch()`, not a query layer.
- **ORM / database layer** ‚Äî Not our job. Use Drizzle, Prisma, Kysely, or raw drivers.
- **Auth system** ‚Äî Basic auth helpers (`pledgestack/auth`) for sessions, OAuth, CSP, SSRF, XSS, audit logging. Not a full identity provider. Use Auth0, Clerk, or your own for complex flows.
- **CMS** ‚Äî Not a CMS. Not a headless CMS. Not a CMS plugin.
- **Email sending** ‚Äî Not built-in. Use Resend, Nodemailer, etc.
- **Queue / job system** ‚Äî Basic `CronScheduler` and `JobQueue` in `pledgestack/api` for simple scheduling. For production queues, use BullMQ, Inngest, etc.
- **Realtime / WebSocket server** ‚Äî `pledgestack-ws` provides WebSocket route support. Not a full pub/sub system.
- **File upload handling** ‚Äî `pledgestack/api` provides `handleUpload` with magic number validation and size limits. For large-scale uploads, use a third-party service.
- **Image optimization CDN** ‚Äî Not building a CDN. PledgePack's asset pipeline handles hashing and format, not on-the-fly optimization.
- **Analytics** ‚Äî Not built-in. Use Vercel Analytics, Plausible, PostHog, etc.
- **A/B testing framework** ‚Äî Not built-in. Cookie-based cache variants enable this, but we don't ship a testing UI.
- **Feature flag system** ‚Äî Not built-in. Use LaunchDarkly, PostHog, or env vars.
- **State management library** ‚Äî `pledgestack/state` provides built-in primitives (`createStore`, `useUrlState`, `useCrossTabState`, `useFormState`, `useOptimisticState`). For complex apps, Zustand/Jotai/Redux still work great alongside it.
- **Component library** ‚Äî Not shipping UI components. Use shadcn/ui, Radix, or your own.
- **Mobile / React Native** ‚Äî Not a mobile framework. Web only.
- **Static export to non-HTML** ‚Äî Not generating PDFs, XML, etc. from routes. Use API routes for that.

## We Don't Need (From Next.js)

- **Pages router** ‚Äî App directory only. No `pages/` directory. No migration path. Clean break.
- **`getServerSideProps` / `getStaticProps`** ‚Äî Replaced by `generateMetadata`, server components, and route segment config.
- **`_app.tsx` / `_document.tsx`** ‚Äî Replaced by root `layout.tsx`.
- **Custom `_error.tsx`** ‚Äî Replaced by `error.tsx` per segment.
- **API routes in `pages/api/`** ‚Äî Replaced by `route.ts` in app directory.
- **`next/legacy/image`** ‚Äî No legacy anything.
- **Internationalization (i18n) config in framework** ‚Äî `pledgestack/a11y` provides RTL support (`useRtl`), translation extraction (`extractTranslations`), and ICU message validation. For full i18n, use `next-intl`, `i18next`, or your own. Middleware handles locale routing.
- **AMP support** ‚Äî Dead technology. Not supporting it.
- **Edge Middleware with Node.js APIs** ‚Äî Middleware runs in a constrained runtime. No `fs`, no `child_process`. This is by design.

## We Don't Need (From Other Frameworks)

- **File-based routing with config files per route** (Remix `loader`/`action` exports) ‚Äî We use `page.tsx` + `route.ts` conventions, not co-located loaders.
- **HTML-first / progressive enhancement** (Remix/Qwik) ‚Äî We're a React framework. Client-side hydration is the default, not progressive enhancement.
- **Resumability** (Qwik) ‚Äî Interesting concept but requires a completely different rendering model. Not compatible with React's hydration model.
- **Signals-based reactivity** (Solid/Qwik) ‚Äî We use React's reactivity. No signals.
- **No virtual DOM** (Solid/Svelte) ‚Äî We use React. Virtual DOM is part of the deal.
- **Compile-time framework** (Svelte/Astro Pledges) ‚Äî We're a runtime framework. PledgePack compiles bundles, not components.
- **Multi-framework support** (Astro) ‚Äî React only. No Vue, Svelte, or Solid components.
- **Zero-JS by default** (Astro) ‚Äî We ship JS. React needs it. We minimize it, but we don't pretend to be static.
